# ADR-003: Redis for Low-Latency Serving

## Context
The API `/api/sentiment` is heavily read by the frontend dashboard. Querying PostgreSQL on every page load adds unnecessary database load for data that only changes when the background ingestion pipeline runs.

## Decision
We implemented a **Redis Cache Layer** to sit in front of PostgreSQL. 

## Rationale
- **Performance**: Redis serves responses in memory, driving API latency down to `~20-50ms`.
- **Scaling**: A caching layer allows the API to handle high request volumes (e.g. from many concurrent dashboard users) without bottlenecking the relational database.
- **Data Freshness**: The cache TTL is configured to match the scheduler interval (e.g., 15 minutes). This ensures users always see the latest snapshot without ever triggering heavy DB operations themselves.

## Consequences
- Requires Redis as a deployment dependency (handled via Docker Compose/Kubernetes).
- Introduces cache invalidation logic, which is kept simple here via TTL expiry.
