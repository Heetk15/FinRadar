# ADR-001: APScheduler vs. Celery

## Context
FinRadar requires a background task to periodically fetch financial news and run NLP sentiment scoring.

## Decision
We chose **APScheduler** running within the FastAPI lifespan over **Celery**.

## Rationale
- **Simplicity**: Celery requires deploying and managing a separate worker container and a message broker (like RabbitMQ or Redis as a broker).
- **Overhead**: Our ingestion pipeline runs sequentially at a fixed interval (e.g., every 15 minutes). We do not currently need highly distributed horizontal scaling for job execution, which is Celery's primary strength.
- **Resource Constraints**: Running a lightweight asyncio scheduler inside the existing FastAPI container reduces memory footprint and simplifies the Docker Compose setup.

## Consequences
- We save infrastructure complexity and reduce deployment points of failure.
- If we ever need to distribute ingestion tasks across multiple worker nodes to bypass rate limits or handle thousands of tickers, we will need to migrate to Celery or Temporal. For the current scope of ~10-50 tickers, APScheduler is perfectly sufficient.
