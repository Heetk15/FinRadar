# ADR-002: Lineage-Aware Snapshot Architecture

## Context
We need to store the historical sentiment of stocks. A naive approach would be to calculate the sentiment dynamically on every request from raw news articles.

## Decision
We chose a **Snapshot Architecture**. Every time the ingestion pipeline runs, it calculates the panic score for a ticker and stores it as a `SentimentSnapshot`. We also maintain a bridge table (`SnapshotArticle`) linking the snapshot to the specific `NewsArticle` rows used to calculate that score.

## Rationale
- **Performance**: Pre-calculating and storing the score drops read latency from tens of seconds to milliseconds.
- **Data Lineage / Auditing**: If an anomaly is detected in the historical sentiment graph, we can trace exactly which news headlines contributed to that specific panic score at that specific point in time. 
- **Idempotency**: Articles can be ingested repeatedly without skewing historical data, since the snapshot freezes the calculation.

## Consequences
- Requires a slightly more complex data model (Bridge table) and uses more database storage than simply keeping the raw articles.
- Yields a massive performance and traceability improvement.
